### Q1（设计题 - 代理模式：商品缓存）
- **题目类型**：设计题
- **难度**：L2
- **问题**：商品详情接口 QPS 极高，数据更新频率低，直接在业务方法里加缓存 if-else 有什么问题？如何分离缓存与业务逻辑？
- **学生回答**：代理模式。缓存逻辑在代理实现类中（支持 local/redis 可选），业务方法加注解，用 AOP 切面实现代理。调用方完全无感知，不需要改。
- **评判结果**：✅ 正确
- **得分**：9/10
- **评分明细**：
  - 正确性：3/3（代理模式准确，AOP + 注解是标准实现）
  - 深度：2/3（未显式指出原方案违反 SRP）
  - 广度：2/2（考虑了缓存实现可替换、调用方无感知）
  - 实战感：2/2（AOP + 注解 = Spring @Cacheable）
- **反馈**：选型准确。补充：直接在业务方法里加缓存违反单一职责原则（SRP），业务方法同时承担「获取数据」和「管理缓存」两个职责。代理模式将两者物理隔离。
- **知识点标签**：代理模式, AOP, SRP, Spring @Cacheable

### Q2（概念题 - 代理 vs 装饰器）
- **题目类型**：概念题
- **难度**：L2
- **问题**：代理和装饰器代码结构几乎一样，本质区别是什么？
- **学生回答**：装饰器的语义是对原有逻辑的增强/补充，围绕原业务。代理的语义是对原有逻辑的控制，隐藏真实对象的访问和细节，可添加额外功能。
- **评判结果**：✅ 正确
- **得分**：9/10
- **评分明细**：
  - 正确性：3/3（「增强 vs 控制」抓住核心语义差异）
  - 深度：2/3（可补充行为推论：装饰器一定转发，代理可以拦截/跳过）
  - 广度：2/2
  - 实战感：2/2
- **反馈**：核心判断精准。补充行为推论：装饰器一定调用原对象（增强前提是原逻辑执行），代理不一定调（缓存命中直接返回，不调原方法）。
- **知识点标签**：代理vs装饰器, 增强vs控制, 组合模式语义

### Q3（设计题 - CQRS：读写分离）
- **题目类型**：设计题
- **难度**：L2
- **问题**：订单服务读占90%写占10%，读写共享模型互相影响，如何把读写拆开设计？读写模型数据怎么同步？
- **学生回答**：1）Redis 分布式锁 + 预占库存替代行锁优化写操作；2）MySQL 主从同步做读写分离（主写从读、GTID 模式、多从库负载均衡）。
- **评判结果**：⚠️ 部分正确
- **得分**：6/10
- **评分明细**：
  - 正确性：1/3（描述的是 DB 层读写分离，非应用模型层面的 CQRS）
  - 深度：2/3（分布式锁+预占库存细节实战）
  - 广度：1/2（只关注基础设施层，漏应用模型维度）
  - 实战感：2/2（Redis 锁 + 主从同步是生产级方案）
- **反馈**：DB 主从分离解决的是"读写抢锁"，但从库表结构和主库一样，多表 JOIN 依然慢。CQRS 核心是模型分离：写模型范式化保证一致性，读模型反范式化（宽表/预聚合）零 JOIN 查询。同步方式：领域事件 / CDC（Canal）。
- **知识点标签**：CQRS, 读写模型分离, DB读写分离vs模型分离

### Q4（追问 - CQRS 延迟问题）
- **题目类型**：场景题
- **难度**：L2
- **问题**：写模型发事件→消费端更新读模型有延迟，用户刚下单就刷列表看不到新订单怎么处理？
- **学生回答**：1）用 Canal 等 CDC 方式写入宽表做模型分离（补充上题）；2）写库成功后通过异步/MQ 或 AOP 同步将宽表数据写入读缓存，达到近零延迟；极端情况可接受。
- **评判结果**：⚠️ 部分正确
- **得分**：7/10
- **评分明细**：
  - 正确性：2/3（Canal/CDC 有效，但「写侧直接写读缓存」破坏读写解耦）
  - 深度：2/3（实战做法没问题，缺 read-your-own-writes 标准模式）
  - 广度：1/2（未考虑客户端乐观展示、读路由降级策略）
  - 实战感：2/2
- **反馈**：标准做法是 Read-your-own-writes：①客户端乐观展示（前端用返回数据拼假记录）；②写后读主库（带版本号，读模型落后时回退查主库）；③Session 级路由（写后 N 秒内读请求打主库）。避免写侧感知读模型结构。
- **知识点标签**：CQRS延迟, Read-your-own-writes, CDC, Canal

### Q5（设计题 - Event Sourcing：余额系统）
- **题目类型**：设计题
- **难度**：L2
- **问题**：账户余额只存最终值，无法审计追溯、对账、修复 Bug。如何重新设计存储？
- **学生回答**：balance 做冗余字段，每次充值/消费/退款都写表存储谁、什么时间、操作了什么、是否成功。
- **评判结果**：⚠️ 部分正确
- **得分**：7/10
- **评分明细**：
  - 正确性：2/3（核心思路正确：事件是真相，余额是推导值。但未提到事件不可变性和余额可重建性）
  - 深度：2/3
  - 广度：1/2（未回扣 Bug 修复场景）
  - 实战感：2/2
- **反馈**：核心正确。ES 最值钱的属性是「快照可从事件流重建」：Bug 写坏余额→删快照→修 Bug→从事件 replay→得到正确余额。定期打 Snapshot 解决 replay 性能问题。
- **知识点标签**：Event Sourcing, Snapshot, 事件不可变, 状态可重建

### Q6（追问 - ES Snapshot 性能优化）
- **题目类型**：场景题
- **难度**：L2
- **问题**：用户有 50 万条事件，查余额每次全量 replay 吗？怎么优化？
- **学生回答**：对余额做冗余字段，事务内同步增减，查询只查此字段。详细记录按时间筛选。
- **评判结果**：⚠️ 部分正确
- **得分**：7/10
- **评分明细**：
  - 正确性：2/3（做法可行，但冗余余额独立更新不可从事件重建，非 ES Snapshot）
  - 深度：2/3
  - 广度：1/2（未回扣 Bug 写坏余额后如何修复）
  - 实战感：2/2
- **反馈**：ES Snapshot 与「冗余余额」的关键区别：Snapshot 可从事件流重建（删除→replay→恢复正确值），冗余余额独立更新不可重建。标准做法：每 N 条事件打一次 Snapshot，查余额 = 最新 Snapshot + replay 之后的事件。
- **知识点标签**：Snapshot, 可重建性, Event Sourcing性能优化

- **本次答题数**：6

- **关键收获**：

- **薄弱环节**：
  - DB 读写分离 vs 模型读写分离的层次区分

- **建议下次复习**：CQRS 模型分离 + Read-your-own-writes；ES Snapshot 可重建语义
