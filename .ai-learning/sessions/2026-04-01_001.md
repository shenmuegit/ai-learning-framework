## 第3节：TCC 三坑（讲解）

### 1. 三个坑分别是什么
1. `幂等`：同一阶段的 `Try/Confirm/Cancel` 重复调用只能生效一次。
2. `空回滚`：`Cancel` 先到，但本地根本没有成功的 `Try` 记录。
3. `悬挂`：`Cancel` 已执行，结果一个迟到的 `Try` 又跑进来冻结资源。

### 2. 为什么它们容易混
1. 三者都和网络重试、超时、乱序有关。
2. 但根因不同：幂等解决“重复执行”，空回滚/悬挂解决“时序错乱 + 状态识别”。

## 第3节追问同步

- **学生回答摘要**：
  1. 正确认出“`Cancel` 先到且本地查不到 `Try` 记录”叫 **空回滚**，并指出不能直接改余额字段，而应基于事务记录处理；
  2. 接受修正：空回滚时即便没有 `Try` 记录，也要落一条 `CANCELED` 标记，防止迟到的 `Try` 再执行成功；
  3. 正确认出“`Cancel` 已完成后迟到 `Try` 又到”叫 **悬挂**，并接受“`Try` 先查回滚标记”的防护口径；
  4. 接受修正：**幂等只能解决重复执行，空回滚/悬挂本质是时序错乱 + 状态识别问题**；
  5. 接受修正：`Confirm` 在 `CANCELED` 状态下不是“重复确认”，而是非法时序，应返回已取消/拒绝确认。
- **评判结果**：✅ 正确（补充后）

  - 正确性：4/4
  - 架构约束：2/2

  - 工程可执行性：2/2

### 三项反馈（追问）

- **正确点**：
  1. 你已经能稳定区分：`空回滚 = 没有成功 Try 却来了 Cancel`，`悬挂 = Cancel 之后迟到 Try 又执行`。
  2. 你接受了最关键的防护动作：`Cancel` 查不到记录也要落 `CANCELED` 标记。
  3. 你已收住 `Confirm` 的状态判断：`CONFIRMED` 才是重复确认，`CANCELED -> Confirm` 必须拒绝。
- **风险点**：
  1. 幂等键当前仍偏粗，复杂履约场景下不能长期只依赖 `order_id`。
  2. 事务状态枚举仍可继续收敛为 `FROZEN / CONFIRMED / CANCELED`，减少“未冻结/已取消/已冻结”混用。
- **最小修正建议**：
  1. 以后答题固定写：`幂等管重复，状态表管顺序。`
  2. 事务表优先围绕“幂等判断、状态流转、空回滚标记、悬挂拦截”设计，不要先堆业务字段。

## 第3节应用题（收官小测）

- **目标**：验证你是否能把 `Try -> Cancel -> 迟到 Try` 的复杂时序压成稳定口径。
- **题目**：
  账户服务 `TCC` 冻结余额，发生如下时序：
  1. 协调器调用 `Try(order_id=123, amount=100)`
  2. 账户服务实际执行成功，冻结了 100 元，但响应在网络里丢了
  3. 协调器以为失败，于是调用 `Cancel(order_id=123)`
  4. `Cancel` 先到并执行
  5. 随后网络里那个迟到的 `Try` 请求又到了一次

  请按 3 段回答：
  1. `这里同时涉及哪两个坑`
  2. `正确处理顺序`
  3. `最终余额和事务状态应该是什么`

## 第3节作答同步（小测）

- **学生回答摘要**：
  1. 判断该场景涉及 **悬挂** 与 **幂等状态判断**；
  2. 给出顺序：第一次 `Try` 成功冻结，`Cancel` 回滚释放，迟到的 `Try` 因已取消而失败；
  3. 最终结论：余额应回到冻结前，事务状态为 `已取消`；
  4. 补充修正：该题不是空回滚，因为第一次 `Try` 实际已经成功落地。
- **评判结果**：✅ 正确
- **得分**：9/10

### 三项反馈（小测）

- **正确点**：
  1. 你已经能把“第一次 `Try` 真成功”和“协调器误判失败”区分开，不会误判为空回滚。
  2. 你明确了最终效果：只释放一次 100 元，迟到的 `Try` 必须被 `CANCELED` 标记拦截。
  3. 你已收住最关键句式：`空回滚是没有成功 Try 却来了 Cancel；悬挂是 Cancel 之后迟到 Try 又来了。`
- **风险点**：
  1. 事务表最小字段设计还可再继续压缩成“必需字段清单”。
  2. `Confirm`、`Cancel` 的状态转移口径在代码设计题里仍要继续练熟。
- **最小修正建议**：
  1. 后续若做代码题，优先写状态机分支而不是先写 SQL。
  2. 固定一句：`CANCELED -> Confirm` 是非法时序，不是重复确认。
