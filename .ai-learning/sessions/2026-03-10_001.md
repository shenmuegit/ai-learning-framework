## 问答记录

### Q1（L2 设计题 - 支付成功但库存扣减失败的补偿链路）
- **题目类型**：设计题
- **难度**：L2
- **问题**：请设计“支付成功但库存扣减失败”的 Saga 补偿流程，覆盖状态流转、退款触发、预占释放、终态收敛。
- **学生回答摘要**：
  - 给出两类业务结果：最终库存扣减成功 / 自动退款；
  - 支付成功判断包含回调、主动轮询、客户端查询；
  - 采用 MQ 重试、死信、幂等键与人工兜底；
  - 补充了订单、支付状态流转与退款处理思路。
- **评判结果**：✅ 正确（多次补充后）
- **得分**：9/10（补充后）
- **评分明细**：
  - 正确性：2/3（核心链路已齐全，但预占释放SQL语义仍需校正）
  - 深度：3/3（已覆盖阈值重试、自动退款、预占释放、DB幂等记录）
  - 架构约束：2/2（已澄清最终一致性基于 MQ + 幂等，不依赖跨服务强事务回滚）
  - 实战感：2/2（幂等、重试、死信、告警具备工程落地性）
- **反馈（关键纠偏）**：
  1. 订单服务与库存服务是跨服务关系，不能依赖“一个事务回滚全部”；应使用本地事务 + 事件驱动补偿。
  2. “消息进队列就不需要补偿”不成立；仍需 Outbox/去重与重放策略保障一致性。
  3. 推荐终态状态机：
     - 订单：`PENDING_PAY -> PAID -> FULFILL_FAILED -> REFUNDING -> CANCELED`
     - 支付：`PAY_SUCCESS -> REFUNDING -> REFUNDED`
     - 库存预占：`RESERVED -> DEDUCT_FAILED -> RELEASED`
  4. 标准补偿链路（示例）：
     - 步骤1：支付回调确认成功 -> 写支付成功本地事务 + 发 `PaymentSucceeded`
     - 步骤2：库存消费 `PaymentSucceeded` 扣减失败 -> 标记 `DEDUCT_FAILED` + 发 `InventoryDeductFailed`
     - 步骤3：订单消费失败事件 -> 订单置 `FULFILL_FAILED/REFUNDING` + 发 `OrderRefundRequested`
     - 步骤4：支付消费退款请求 -> 调第三方退款成功后发 `RefundSucceeded`
     - 步骤5：库存消费退款成功事件 -> 释放预占并记流水
     - 步骤6：订单消费退款成功事件 -> 终态置 `CANCELED`，闭环审计
  5. 幂等键建议：`payment_order_id`、`reservation_id`、`refund_request_id`、`event_id`。
- **学生补充（追答）**：
  - “同步”描述是业务路径差异，不是跨服务强事务；
  - 跨事务收敛依赖最终一致性；
  - 最终一致性依赖 MQ + 幂等。
- **导师复评**：
  - 该澄清修正了最关键的架构边界误解，架构约束项上调为满分；
  - 仍需把补偿链路写成可执行步骤（触发条件、动作、补偿、幂等键）以达到 9~10 分。
- **学生补充（二次追答）**：
  - 消费端异常时通过 RabbitMQ `basicNack(requeue=true)` 触发重投（手动 ack）；
  - 消费前用 Redis 基于 `orderId` 做幂等标记，成功后推进订单状态，异常则重试；
  - 目标终态：订单已支付、支付已支付、库存扣减正确（含预占扣减）。
- **导师二次复评**：
  - 该补充提升了“消费重试 + 幂等防重”的可操作性，工程实现更清晰；
  - 但仍缺少“重试上限后的自动退款分支”和“预占释放分支”的显式步骤，因此总分维持 `8/10`；
  - 另外，幂等标记不建议简单“异常即删除”，应结合状态机/去重表避免并发抖动导致重复消费窗口。
- **学生补充（三次追答）**：
  - 新增“重试上限后自动退款”分支，并给出定时扫描触发退款思路；
  - 新增库存服务释放预占逻辑；
  - 幂等方案升级为“DB消费记录表 + 缓存幂等标记兜底”。
- **导师三次复评**：
  - 本次补充补齐了上一轮的两大缺口，分数上调为 `9/10`；
  - 关键纠偏：若 `inventory_quantity_prep` 表示“已预占量”，释放预占应执行减法（`- quantity`），并配合乐观锁版本号或条件更新防并发错扣；
  - “暂停其它队列”建议收敛为“按订单维度熔断/隔离”，避免系统级联阻塞。
- **知识点标签**：Saga, 补偿事务, 本地事务+事件, Outbox, 幂等与重试
