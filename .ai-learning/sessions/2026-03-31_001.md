## 第2节应用题（1题1目标）

- **目标**：为 `MySQL + Redis + 第三方支付 + 优惠券服务` 的交易链路选择合适的分布式事务模型。
- **题目**：
  你有一个跨服务交易链路：
  1. 订单库：`MySQL`
  2. 库存缓存：`Redis`
  3. 支付通道：第三方网关
  4. 优惠券：独立优惠券服务

  请回答：
  1. `资源类型判断`
  2. `XA不适配点`
  3. `更合适的事务模型`
  4. `代价与前提`

---

## 第2节作答同步

- **学生回答摘要**：
  1. 判断该链路涉及多异构资源，尤其包含 `Redis` 与第三方支付网关；
  2. 指出 `XA` 不适配点包括：异构资源难接入、可用性差、高流量场景性能不佳；
  3. 选择 `Saga + TCC`，并说明其更符合预留资源、最终一致、异常恢复的现实需求；
  4. 在后续追问中给出补偿优先级：
     - 支付成功但优惠券确认失败：订单保持成功，优惠券走补偿/重试分支；
     - 优惠券失败通常比余额失败更容易接受补偿；
     - 库存失败通常也比余额失败更容易接受补偿，因为余额更接近资金事实。
- **评判结果**：✅ 正确（补充后）
- **得分**：9/10
- **评分明细**：
  - 正确性：4/4
  - 架构约束：2/2
  - 完整性：1/2（对“优惠券补偿”还需避免默认直接补发券）
  - 工程可执行性：2/2

### 三项反馈

- **正确点**：
  1. 你已经明确：第三方支付这类外部副作用无法像数据库事务那样被 `XA` 统一回滚。
  2. 你能把“整体最终一致”与“关键资源强约束保护”分开表达，而不是把 `Saga + TCC` 混成一个模糊名词。
  3. 你已建立补偿优先级：资金事实 > 内部资源 > 附属权益。
- **风险点**：
  1. “优惠券失败就补发券”不能作为默认口径，需先重试核销/确认已用，再按运营规则决定是否补偿权益。
  2. “Saga + TCC 最优”表述过强，更准确是“在这类异构资源链路里更现实、更可落地”。
- **最小修正建议**：
  1. 固定一句：`整体流程可走 Saga 编排，关键资源用 TCC 风格预留与确认/释放。`
  2. 固定一句：`支付成功是外部资金事实，通常不因优惠券确认失败而回滚整单。`

## 第2节结论

- **收官结果**：✅ 通过
- **最终得分**：9/10
- **固定口径**：
  1. `异构资源链路不适合优先选 XA/2PC，因为并非所有资源都能可靠参与同一个全局提交协议。`
  2. `第三方支付一旦扣款成功，通常应保持主交易成功，失败的优惠券、库存等侧资源进入补偿分支。`
  3. `余额/支付更接近资金事实，补偿空间比库存和优惠券更小。`
