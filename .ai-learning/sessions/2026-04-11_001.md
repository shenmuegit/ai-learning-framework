## 第2节：回查与编排语义（多题串讲）

### 学生作答摘要

1. 给出回查响应基础结构，并坚持 `code-only` 语义；
2. 逐步接受关键边界：命中 `PROCESSING` 需复用旧 `request_id`，不得换键重试；
3. 定义关键码值：`1001`（处理中）、`1002`（超时失败）、`1003`（回查未命中）、`2001`（网关参数缺失）、`200`（成功）、`500`（系统异常）；
4. 完成客户端动作闭环：`1001` 固定间隔回查 -> 超时后 `1002` 新键重试 -> `1003` 自动新键重试一次后提示用户；
5. 接受安全补充：回查应使用 token 身份校验并校验 owner，`401/403` 语义区分应一致。

### 三项反馈

- **正确点**：
  1. 你把“可等、可查、不可换键”从概念推进到了流程规则；
  2. 你能把多个业务码值串成统一客户端动作，具备编排思维；
  3. 你在安全与体验之间找到了折中方案（鉴权 + owner 校验 + 失败后自动一次重试）。
- **风险点**：
  1. `code-only` 方案仍需长期治理纪律，新增码值时要保证分层不漂移；
  2. `1003` 自动重试策略应设置全局上限与去抖，避免边缘场景下重试风暴。
- **最小修正建议**：
  1. 固化码位模板（如 `MMSSXXX`）并建立变更评审；
  2. 固化回查接口最小协议：`request_id -> code, result`，并对外文档明确每个 code 的客户端动作。
