## 第1节：库存 RT 抖动定位（讲解）

### 1. 谁会触发库存服务调用
1. 订单服务：下单阶段同步调用库存服务，执行“预占库存”校验与写入。
2. 库存消费者：支付成功后异步执行“预占转实扣”。
3. 取消/退款链路：异步执行“释放预占”。
4. 运营与查询链路：库存查询、对账、修复任务也可能触发读写。

### 2. 订单服务如何调用库存服务
1. 下单主链路：同步 RPC（或 HTTP）调用库存的 reserve 接口。
2. 支付后链路：通过 MQ 事件触发库存消费者异步处理。
3. 失败补偿链路：通过 MQ/定时任务回查触发释放动作。

### 3. 调用结果是什么（预占还是实扣）
1. 下单同步调用返回的是“预占结果”（成功/失败与原因），不是实扣结果。
2. 实扣发生在“支付成功后的异步阶段”。
3. 若超时取消或退款，走“释放预占”而不是实扣。

### 4. 为什么会出现库存 RT 抖动
1. 读写混部导致锁竞争：预占、实扣、释放集中在热点 SKU。
2. 库存行锁与连接池争用：事务持锁时间长会放大排队。
3. 重试风暴：超时重试叠加导致瞬时流量放大。
4. 消费积压回放：MQ 堆积后集中重放会形成写峰值。

## 第2节：预占/实扣/释放 三动作边界与一致性时序图（讲解）

### 1. 三动作定义（必须分层）
1. `预占`：在下单阶段执行，目标是“锁定可售额度”，不改变已售事实。
2. `实扣`：在支付成功后执行，目标是“确认销售事实”，把预占转换为已售。
3. `释放`：在超时取消/支付失败/退款等场景执行，目标是“归还预占额度”。

### 2. 单一事实口径
1. 订单服务负责“订单状态事实”。
2. 库存服务负责“库存状态事实”。
3. 事件只传播“状态变化”，不直接篡改对方事实。

### 3. 推荐状态机（库存侧）
1. `PENDING(未预占)` -> `RESERVED(已预占)` -> `DEDUCTED(已实扣)`
2. `RESERVED(已预占)` -> `RELEASED(已释放)`
3. 任何回滚/重试都不能跳过状态机前置条件。

### 4. 一致性时序（最小口径）
1. 下单：订单创建待支付 -> 触发预占 -> 返回预占结果。
2. 支付成功：支付回调成功 -> 触发实扣 -> 订单进入待发货。
3. 超时取消：超时任务触发 -> 触发释放 -> 订单进入已取消。
