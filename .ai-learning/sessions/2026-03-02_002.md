## 问答记录

### Q1（设计题 - 多支付渠道重构）
- **题目类型**：设计题
- **难度**：L2
- **问题**：多支付渠道（微信/支付宝/银联/Apple Pay）各自签名、报文、回调不同，且每季度新增 1~2 渠道，部分渠道有 v1/v2 并存。如何用设计模式重构 if-else 代码？
- **学生回答**：单例 + 策略 + 适配器。策略模式每个渠道一个实现；单例无状态省内存免并发；适配器统一入参出参。新增渠道只需实现策略接口。
- **评判结果**：⚠️ 部分正确
- **得分**：7/10
- **评分明细**：
  - 正确性：2/3（策略正确，缺工厂/注册表，适配器定位偏差）
  - 深度：2/3（单例论述好，漏了 channel→策略 的路由）
  - 广度：2/2
  - 实战感：1/2（未回应 v1/v2 并存）
- **反馈**：策略选得准。关键缺失：需要工厂+注册表（如 Map<String, PayStrategy>）来消除路由处的 if-else。适配器更适用于包装无法修改的第三方 SDK，而非统一自己的入参出参（策略接口本身就该定义统一模型）。
- **知识点标签**：策略模式, 工厂模式, 注册表, 适配器模式, 开闭原则

### Q2（追问 - v1/v2 灰度流量切换）
- **题目类型**：设计题
- **难度**：L2
- **问题**：微信支付 v1(XML) 和 v2(JSON) 并存，灰度 10% 走 v2，如何在策略体系里处理？
- **学生回答**：接口加版本号区分实现；工厂由 version+channel 决定注入哪个实现；网关层改路由切流量，业务无关可动态配置。
- **评判结果**：⚠️ 部分正确
- **得分**：7/10
- **评分明细**：
  - 正确性：2/3（version+channel 组合键对，网关路由层次偏差）
  - 深度：2/3（考虑了生命周期管理）
  - 广度：2/2
  - 实战感：1/2（网关适合入站路由，出站版本选择应在工厂内部）
- **反馈**：version+channel 组合键正确。但版本差异是出站调用（调第三方 API 用 XML 还是 JSON），不是入站请求差异，应在策略工厂内部按配置中心比例路由，而非网关层。判断标准：入站路由→网关，出站策略→工厂内部。
- **知识点标签**：工厂路由, 灰度发布, 配置中心, 入站vs出站路由

### Q3（设计题 - 通知服务）
- **题目类型**：设计题
- **难度**：L2
- **问题**：通知服务支持短信/邮件/App推送/站内信，同一事件可触发多渠道，用户可自定义订阅。用什么模式？与支付场景策略模式的本质区别？
- **学生回答**：责任链 + 工厂 + 单例。每一链决定参数交由工厂生成渠道实现。真实场景偏向 MQ + 消费者调用策略。
- **评判结果**：⚠️ 部分正确
- **得分**：7/10
- **评分明细**：
  - 正确性：1/3（责任链不是最佳选择，漏了观察者模式）
  - 深度：2/3（MQ 方案有深度）
  - 广度：2/2
  - 实战感：2/2（MQ + 消费端策略非常实战）
- **反馈**：核心特征是「1个事件 → N个渠道同时通知」，这是观察者/发布-订阅模式的语义，非责任链（1:1 链式传递）。MQ 本身就是 pub-sub 的基础设施实现，代码设计层应用观察者模式名称。区分标准：1:1路由→策略，1:N广播→观察者，链式过滤→责任链。
- **知识点标签**：观察者模式, 发布-订阅, 责任链vs观察者, 事件驱动

### Q4（追问 - 用户订阅过滤层次）
- **题目类型**：设计题
- **难度**：L2
- **问题**：用户自定义订阅的过滤逻辑放在发布端、MQ 层还是消费端？为什么？
- **学生回答**：消费端。理由：1）不阻塞发布端业务；2）MQ 只负责广播不耦合业务逻辑。补充：消费端分两层——过滤层 + 发送渠道实现层，解耦过滤和发送。
- **评判结果**：✅ 正确
- **得分**：10/10
- **评分明细**：
  - 正确性：3/3
  - 深度：3/3（两层消费端设计体现清晰分层思维）
  - 广度：2/2
  - 实战感：2/2
- **反馈**：三层职责划分（发布端fire-and-forget / MQ纯广播 / 消费端过滤+发送）非常干净。两层消费端设计将过滤逻辑与发送逻辑拆开，各自可独立演进。极端场景下可在发布端做粗粒度路由（按渠道类型分Topic）节省MQ吞吐，但常规业务量下此方案完全正确。
- **知识点标签**：消费端过滤, 职责分层, 发布-订阅, 关注点分离
