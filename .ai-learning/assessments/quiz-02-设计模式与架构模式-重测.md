## 题目与作答

### 第1题（GoF模式 - 日志系统）
- **场景**：日志写入固定流程（格式化→过滤敏感字段→写入目标），目标可扩展，部分日志可叠加压缩/加密
- **学生回答**：观察者模式 + 装饰器模式 + 模板方法模式
- **参考答案要点**：模板方法（固定骨架）+ 装饰器（压缩/加密叠加）+ 观察者或策略（多目标分发/选择）
- **得分**：9/10
- **评语**：三种模式都答到了。模板方法与装饰器准确；观察者在“同时广播到多个目标”语境下成立。

### 第2题（CQRS + Event Sourcing - 保险理赔）
- **场景**：复杂状态流转、监管审计、查询量远大于写入
- **学生回答**：记录每次状态操作明细；读侧采用宽表 + 主从兜底；通过 Canal 订阅 binlog 转换并写入宽表；写后返回版本号，查询按版本回退主库
- **参考答案要点**：写侧事件明细/不可变记录满足追溯；读侧反范式化宽表/时间线视图；通过事件/CDC 同步；用版本号或回退主库处理延迟
- **得分**：9/10
- **评语**：已纠正“宽表不能靠主从复制同步”的问题，Canal + 宽表 + 版本兜底回答完整。若再明确“事件 append-only 不可变”会更严谨。

### 第3题（DDD逻辑归属 - 酒店预订）
- **场景**：A入住日期校验 / B同房间冲突检查 / C发短信
- **学生回答**：A实体内部；B领域服务；C应用服务编排，出站适配器实现
- **参考答案**：A实体内部 / B领域服务 / C应用服务
- **得分**：10/10
- **评语**：逻辑归属准确，已正确区分领域服务与应用服务。

### 第4题（六边形架构 - 房间库存）
- **场景**：领域服务需查 Redis 库存 + MySQL 房间信息
- **学生回答**：领域层定义 `InventoryRepository`、`RoomRepository`；基础设施层实现；领域服务依赖接口，接口由实现类完成
- **参考答案要点**：领域层定义端口，基础设施层实现适配器，依赖方向向内
- **得分**：9/10
- **评语**：结构清晰，依赖方向正确。端口拆分也比之前更清楚。

### 第5题（综合设计 - 退款）
- **场景**：RefundService 800行，含退款校验/多渠道退款/退款后多动作
- **学生回答**：退款条件校验放在 Order 实体；支付 API 调用由应用服务 + 出站适配器实现；多支付渠道用策略模式；退款后恢复库存/发通知/记审计用观察者模式
- **参考答案要点**：实体（退款条件校验）+ 应用服务（编排/调支付）+ 策略（多渠道）+ 观察者/事件（退款后多动作）
- **得分**：9/10
- **评语**：核心结构已经成型。若再补一句“应用服务发布退款完成事件，由观察者各自处理后续动作”会更完整。
